   584→## Round 4 — WebSocket Real-Time Price Streaming
   585→
   586→**Task ID:** 2
   587→**Agent:** fullstack-developer
   588→**Date:** 2025-01-xx
   589→
   590→### What Was Built
   591→
   592→A **WebSocket mini-service** using Socket.io that streams real-time crypto price updates to the frontend, replacing polling-only market data with live-streamed data.
   593→
   594→#### 1. Mini-Service (`mini-services/price-stream/`)
   595→- **`package.json`**: Independent bun project with `socket.io` dependency, port 3003, `bun --hot` dev script
   596→- **`index.ts`**: Socket.io server on port 3003 that:
   597→  - Fetches BTC/ETH/SOL prices from CoinGecko every 15 seconds
   598→  - Fetches Fear & Greed Index from Alternative.me
   599→  - Emits `price-update` events with `{ prices, updatedAt }` payload
   600→  - Emits `fear-greed-update` events with sentiment data
   601→  - Sends cached data immediately on client connect (zero-latency first paint)
   602→  - Handles graceful shutdown (SIGTERM/SIGINT)
   603→  - Logs connections, disconnections, and price fetches
   604→  - CORS enabled for all origins
   605→  - Uses `path: "/"` for Caddy gateway compatibility
   606→
   607→#### 2. React Hook (`src/hooks/use-price-stream.ts`)
   608→- Connects via `io("/?XTransformPort=3003")` (relative path + query param, NOT direct URL)
   609→- Listens for `price-update` and `fear-greed-update` events
   610→- Returns `{ prices, lastPriceUpdate, fearGreed, connectionStatus, reconnect }`
   611→- Exports TypeScript interfaces: `StreamPrice`, `StreamPriceUpdate`, `StreamFearGreedUpdate`, `ConnectionStatus`
   612→- Auto-reconnects via Socket.io built-in reconnection
   613→- Cleans up socket on unmount
   614→- Manual reconnect via `reconnect()` (increments state counter to re-trigger effect)
   615→
   616→#### 3. Dashboard Integration (`src/components/investor/dashboard.tsx`)
   617→- Imported and used `usePriceStream` hook
   618→- Added **LIVE indicator** in the fund overview banner:
   619→  - Green pulsing dot + "LIVE" when WebSocket connected with data
   620→  - Amber pulsing dot + "CONNECTING" during connection
   621→  - Red pulsing dot + "OFFLINE" when disconnected
   622→- Added **LIVE/DELAYED indicator** in Market Intelligence section header
   623→- Created merged data variables:
   624→  - `livePrices` = stream prices (if available) → fall back to polled API prices
   625→  - `liveSentiment` = stream fear/greed → fall back to polled API sentiment
   626→  - `isLive` = `connectionStatus === "connected" && streamPrices.length > 0`
   627→- Updated all market data consumers to use merged data:
   628→  - Market Ticker Bar (scrolling prices + F&G badge)
   629→  - Market Intelligence cards (BTC/ETH/SOL price cards)
   630→  - Market Sentiment gauge (Fear & Greed + BTC Dominance)
   631→
   632→### Files Created
   633→- `mini-services/price-stream/package.json` — new
   634→- `mini-services/price-stream/index.ts` — new
   635→- `src/hooks/use-price-stream.ts` — new

   638→- `src/components/investor/dashboard.tsx` — added usePriceStream hook, LIVE indicator, merged data
   639→- `package.json` — added `socket.io-client` dependency

   641→### Key Architecture Decisions
   642→- **Fallback pattern**: Stream data is preferred but the dashboard gracefully falls back to polled REST API data when WebSocket is disconnected. This ensures the UI always shows data.
   643→- **State-based reconnect**: Uses a `reconnectId` state counter instead of a ref to trigger effect re-runs, satisfying React linting rules.
   644→- **Immediate cache on connect**: Server sends cached prices on connect so clients don't wait 15 seconds for first data.

   970→### Key Architecture Decisions
   971→- **Broadcast efficiency**: Broadcast messages are stored as a single row with `isBroadcast=true` and `recipientId=null`, rather than duplicating for each investor. The investor API queries both broadcast messages and direct messages in a single `OR` query.
   972→- **Auto-read on fetch**: The investor messages API automatically marks individual unread messages as read when fetched, avoiding a separate "mark as read" API call. Broadcast read status is tracked separately.
   973→- **Health aggregation**: The overall health status is computed from 5 checks (DB healthy, DB latency, API latency, price feed, cache), giving "healthy" if all pass, "degraded" if ≥3 pass, "critical" otherwise.
   974→- **Activity timeline**: Merges NAV updates, price feed events, and audit logs into a unified timeline for the admin health dashboard, giving a complete operational picture.
   975→- **2FA adoption tracking**: The health endpoint calculates 2FA adoption rate from the User table, providing admins with visibility into security posture.

  1300→- **Architecture decisions:**
  1301→  - **Soft-link for documentId:** Per the task spec, `documentId` is a plain `String?` field on `StatementRequest` rather than a Prisma FK relation. This keeps the schema minimal but means the API routes fetch linked documents separately (via `db.document.findMany({ where: { id: { in: docIds } } })`) and hydrate them with a `Map`. Trade-off: slightly more query overhead but cleaner schema that doesn't require cascading deletes when a Document is removed.
  1302→  - **Shared PDF generation helper:** Extracted the 150-LOC PDF generation logic from the existing `/api/admin/statements/generate` route into `src/lib/statement.ts` so both the direct admin generation flow and the new fulfill-request flow use the exact same code path. Future-proof: any future caller (e.g. a scheduled monthly statement job) can reuse it.
  1303→  - **Dual notification on fulfill:** When an admin fulfills a request, the investor gets BOTH `new_document` (so the bell badge increments + toast says "New document ready") AND `new_message` (so they get a separate toast saying "Statement Request Completed"). This double-notification is intentional — the document event refreshes the Document Vault, the message event explicitly references the request being fulfilled.
  1304→  - **`@prisma/client` as serverExternalPackage:** This is the long-term fix for the recurring "stale PrismaClient class after schema changes" issue documented across multiple prior rounds (8-A had to restart the dev server for the same problem). By marking @prisma/client as external, Turbopack doesn't bundle it, so the runtime always loads the latest generated client from node_modules. The first compile after the config change takes ~5s longer (because Turbopack has to re-evaluate the module graph), but subsequent schema bumps won't require any server restart at all.
