   513→## Round 4 — WebSocket Real-Time Price Streaming
   514→
   515→**Task ID:** 2
   516→**Agent:** fullstack-developer
   517→**Date:** 2025-01-xx
   518→
   519→### What Was Built
   520→
   521→A **WebSocket mini-service** using Socket.io that streams real-time crypto price updates to the frontend, replacing polling-only market data with live-streamed data.
   522→
   523→#### 1. Mini-Service (`mini-services/price-stream/`)
   524→- **`package.json`**: Independent bun project with `socket.io` dependency, port 3003, `bun --hot` dev script
   525→- **`index.ts`**: Socket.io server on port 3003 that:
   526→  - Fetches BTC/ETH/SOL prices from CoinGecko every 15 seconds
   527→  - Fetches Fear & Greed Index from Alternative.me
   528→  - Emits `price-update` events with `{ prices, updatedAt }` payload
   529→  - Emits `fear-greed-update` events with sentiment data
   530→  - Sends cached data immediately on client connect (zero-latency first paint)
   531→  - Handles graceful shutdown (SIGTERM/SIGINT)
   532→  - Logs connections, disconnections, and price fetches
   533→  - CORS enabled for all origins
   534→  - Uses `path: "/"` for Caddy gateway compatibility
   535→
   536→#### 2. React Hook (`src/hooks/use-price-stream.ts`)
   537→- Connects via `io("/?XTransformPort=3003")` (relative path + query param, NOT direct URL)
   538→- Listens for `price-update` and `fear-greed-update` events
   539→- Returns `{ prices, lastPriceUpdate, fearGreed, connectionStatus, reconnect }`
   540→- Exports TypeScript interfaces: `StreamPrice`, `StreamPriceUpdate`, `StreamFearGreedUpdate`, `ConnectionStatus`
   541→- Auto-reconnects via Socket.io built-in reconnection
   542→- Cleans up socket on unmount
   543→- Manual reconnect via `reconnect()` (increments state counter to re-trigger effect)
   544→
   545→#### 3. Dashboard Integration (`src/components/investor/dashboard.tsx`)
   546→- Imported and used `usePriceStream` hook
   547→- Added **LIVE indicator** in the fund overview banner:
   548→  - Green pulsing dot + "LIVE" when WebSocket connected with data
   549→  - Amber pulsing dot + "CONNECTING" during connection
   550→  - Red pulsing dot + "OFFLINE" when disconnected
   551→- Added **LIVE/DELAYED indicator** in Market Intelligence section header
   552→- Created merged data variables:
   553→  - `livePrices` = stream prices (if available) → fall back to polled API prices
   554→  - `liveSentiment` = stream fear/greed → fall back to polled API sentiment
   555→  - `isLive` = `connectionStatus === "connected" && streamPrices.length > 0`
   556→- Updated all market data consumers to use merged data:
   557→  - Market Ticker Bar (scrolling prices + F&G badge)
   558→  - Market Intelligence cards (BTC/ETH/SOL price cards)
   559→  - Market Sentiment gauge (Fear & Greed + BTC Dominance)
   560→
   561→### Files Created
   562→- `mini-services/price-stream/package.json` — new
   563→- `mini-services/price-stream/index.ts` — new
   564→- `src/hooks/use-price-stream.ts` — new
   565→
   566→### Files Modified
   567→- `src/components/investor/dashboard.tsx` — added usePriceStream hook, LIVE indicator, merged data
   568→- `package.json` — added `socket.io-client` dependency
   569→
   570→### Key Architecture Decisions
   571→- **Fallback pattern**: Stream data is preferred but the dashboard gracefully falls back to polled REST API data when WebSocket is disconnected. This ensures the UI always shows data.
   572→- **State-based reconnect**: Uses a `reconnectId` state counter instead of a ref to trigger effect re-runs, satisfying React linting rules.
   573→- **Immediate cache on connect**: Server sends cached prices on connect so clients don't wait 15 seconds for first data.

   576→- Price-stream service running on port 3003
   577→- Accessible via Caddy gateway at `/?XTransformPort=3003`

   899→### Key Architecture Decisions
   900→- **Broadcast efficiency**: Broadcast messages are stored as a single row with `isBroadcast=true` and `recipientId=null`, rather than duplicating for each investor. The investor API queries both broadcast messages and direct messages in a single `OR` query.
   901→- **Auto-read on fetch**: The investor messages API automatically marks individual unread messages as read when fetched, avoiding a separate "mark as read" API call. Broadcast read status is tracked separately.
   902→- **Health aggregation**: The overall health status is computed from 5 checks (DB healthy, DB latency, API latency, price feed, cache), giving "healthy" if all pass, "degraded" if ≥3 pass, "critical" otherwise.
   903→- **Activity timeline**: Merges NAV updates, price feed events, and audit logs into a unified timeline for the admin health dashboard, giving a complete operational picture.
   904→- **2FA adoption tracking**: The health endpoint calculates 2FA adoption rate from the User table, providing admins with visibility into security posture.

  1230→  - **Soft-link for documentId:** Per the task spec, `documentId` is a plain `String?` field on `StatementRequest` rather than a Prisma FK relation. This keeps the schema minimal but means the API routes fetch linked documents separately (via `db.document.findMany({ where: { id: { in: docIds } } })`) and hydrate them with a `Map`. Trade-off: slightly more query overhead but cleaner schema that doesn't require cascading deletes when a Document is removed.
  1231→  - **Shared PDF generation helper:** Extracted the 150-LOC PDF generation logic from the existing `/api/admin/statements/generate` route into `src/lib/statement.ts` so both the direct admin generation flow and the new fulfill-request flow use the exact same code path. Future-proof: any future caller (e.g. a scheduled monthly statement job) can reuse it.

This double-notification is intentional — the document event refreshes the Document Vault, the message event explicitly references the request being fulfilled.
  1233→  - **`@prisma/client` as serverExternalPackage:** This is the long-term fix for the recurring "stale PrismaClient class after schema changes" issue documented across multiple prior rounds (8-A had to restart the dev server for the same problem). By marking @prisma/client as external, Turbopack doesn't bundle it, so the runtime always loads the latest generated client from node_modules. The first compile after the config change takes ~5s longer (because Turbopack has to re-evaluate the module graph), but subsequent schema bumps won't require any server restart at all.
